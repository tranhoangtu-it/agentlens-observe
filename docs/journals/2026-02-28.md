# v0.4.0 Release — TypeScript SDK & Multi-Language Support Achievement

**Date**: 2026-02-28 21:46
**Severity**: Medium (Release milestone)
**Component**: SDK, Publishing, Documentation
**Status**: Completed

## Summary

v0.4.0 shipped with TypeScript SDK (agentlens-observe@0.1.0 on npm), PostgreSQL backend support, updated docs, and Docker tag bumped. All code, docs, packages published within ~30 hours of initial feature commits.

## The Brutal Truth

We're finally past single-language SDK hell. Python-only was always a trap — Node.js ecosystem devs couldn't touch AgentLens, so market reach was fundamentally bottlenecked. TS SDK shipped with **zero prod dependencies** using native AsyncLocalStorage, which is elegant but also means we bet hard on Node 18+ being the floor. If that assumption breaks, we're stuck.

Shipping docs update at the same time was necessary but felt rushed — had to cram TS SDK coverage into existing architecture docs without major restructuring. The changelog got long (85 lines added), and there's real integration debt between Python/TS SDKs now.

## Technical Details

**TypeScript SDK (commit 27021dd)**
- 5 src files: index.ts, tracer.ts, transport.ts, types.ts, cost.ts
- 3 test files with 30 tests (vitest)
- Zero prod deps (native fetch, AsyncLocalStorage)
- Dual output: ESM + CJS via tsup (~11KB each)
- trace() as higher-order function (not decorator — TS decorators still experimental)
- span() with enter()/exit() pattern for manual tracing
- immediate + batch transport modes
- Cost tracking mirrors Python SDK (27 model pricing)

**PostgreSQL Backend (commit e94578d)**
- DATABASE_URL env var support (sqlite3:/// fallback)
- storage.py modified for connection pooling prep
- conftest.py updated (41 lines, +18 ~)
- Enables multi-tenant setup for upcoming feature work

**Publishing Pipeline**
- Docker: tranhoangtu/agentlens:0.4.0 (latest tag)
- npm: agentlens-observe@0.1.0 (public)
- PyPI: agentlens-observe v0.3.0 (already published)
- GitHub: Release page with all 3 artifacts linked

**Docs Updates (commit 5102144)**
- README: Docker tag bumped 0.3.0 → 0.4.0 + quickstart refresh
- project-overview-pdr.md: Added TS SDK section (49 lines, +17 ~)
- system-architecture.md: New "Multi-Language Support" subsection (93 lines, +28 ~)
- development-roadmap.md: PostgreSQL marked done, roadmap phases adjusted
- project-changelog.md: New v0.4.0 entry + TS SDK detail (85 lines, +47 ~)

## What We Tried

1. Started with Python-only SDK — worked great for Python teams, but shut out Node.js market entirely
2. Considered gRPC/protobuf first for cross-language support — too heavyweight for observability, killed that idea
3. Built TS SDK without external deps — AsyncLocalStorage + native fetch is Node's strength; paid off

## Root Cause Analysis (Why This Matters)

**Market Limitation**: Python devs represent ~40% of AI/agent builder mindshare. Node/TS is another 35%. By shipping Python-only, we ignored 35% of addressable market. TS SDK changes that.

**Integration Debt**: We now maintain parallel SDKs. Both need version sync, parity testing, feature parity. Every new feature requires 2x work. No build-once-deploy-everywhere luxury. This is expensive.

**Deployment Surface**: Docker tag needs sync with both SDK versions. Currently manual. Will break eventually when someone publishes TS SDK v0.2 but forgets to update Docker tag.

## Lessons Learned

1. **Multi-SDK Parity = Operational Tax**: Every feature now costs 2x. Need automated parity tests (same test suite runs on both SDKs) before shipping TS v1.0.

2. **Zero Deps Was Right Call**: AsyncLocalStorage + fetch eliminates npm Hell. TS SDK is 11KB; if we used axios + other stuff, would be 150KB+. Kept us nimble.

3. **Docs Lag Real Code**: Updated docs 30 hours AFTER code shipped. Should have documented in parallel. Created 1-2 day window where docs were wrong.

4. **Version Sync Will Bite Us**: PyPI = 0.3.0, npm = 0.1.0, Docker = 0.4.0. Looks intentional, but it's confusing. Next time, all three release at same semver.

## Next Steps

1. **Build Parity Test Suite** (week 1): Write cross-SDK test harness that validates both Python + TS SDKs with same test specs
2. **PostgreSQL Migration** (week 2-3): Default to PostgreSQL for new deploys; optional for existing sqlite3 users
3. **Alerting System** (weeks 4-6): Next roadmap item; needs both SDKs to send anomaly signals
4. **npm/PyPI/Docker CI** (ongoing): Automate version bumping across all 3 repos to prevent sync skew

## Unresolved Questions

- Should TS SDK v1.0 wait for decorator support (TS 5.2+) or ship with HOF pattern as-is?
- How to handle SDK feature divergence if one platform gets a feature the other can't support?
- PostgreSQL: Still using SQLite WAL in tests. When do we migrate test infra to postgres://localhost?

# Alerting System: Behavior Anomaly Detection Shipped

**Date**: 2026-02-28 22:27
**Severity**: High (Complex feature with evaluation engine)
**Component**: Server alert engine, SSE pipeline, Dashboard
**Status**: Resolved

## What Happened

Shipped a full anomaly detection system in one session: 4 anomaly types (cost spike, error rate, latency, missing span), absolute + relative threshold modes, 60-second cooldown, real-time SSE delivery, webhook dispatch, and two dashboard pages. 1,215 lines added across 12 files. 14 new tests, 60 total.

## The Brutal Truth

This was exhausting to get right. Alerting systems look simple until you actually build one. The evaluation engine (`alert_evaluator.py` at 188 lines) is carrying enormous implicit complexity — rolling window aggregation, baseline calculation, cooldown tracking, and anomaly detection logic all crammed into one file. It's already bumping against the 200-line modularization threshold after a single pass. The moment we add anomaly type #5, it splits.

The real pain: `missing_span` detection requires knowing what spans *should* exist, which means we're making assumptions about agent topology that users never actually declared. We're guessing based on historical patterns with zero formal schema. That's a ticking time bomb — a new agent type with different span structure will false-alarm constantly.

## Technical Details

**New files (commit fd45b67):**
- `server/alert_models.py` (86 lines): AlertRule, AlertEvent SQLModel tables; 4 anomaly types as enums
- `server/alert_storage.py` (133 lines): CRUD for rules + events; rolling-window query helpers
- `server/alert_evaluator.py` (188 lines): Core evaluation loop; absolute vs. relative threshold; 60s cooldown via in-memory dict
- `server/alert_notifier.py` (48 lines): SSE broadcast + optional webhook POST
- `server/alert_routes.py` (100 lines): REST CRUD + GET /alerts/stream SSE endpoint
- `dashboard/src/lib/alert-api-client.ts` (121 lines): TypeScript client for alert routes
- `dashboard/src/pages/alert-rules-page.tsx` (174 lines): Rule management UI
- `dashboard/src/pages/alerts-list-page.tsx` (111 lines): Fired alerts list + dismiss

**Key design choices:**
- Cooldown stored in-memory (`dict[str, datetime]`) — not persisted, resets on server restart
- SSE reuses existing `/api/traces/stream` pattern, adds `event: alert_fired` type
- Threshold mode `relative` = percent deviation from 30-trace rolling baseline
- Dashboard sidebar badge via polling `/api/alerts?resolved=false` every 10s

**Test results:** 14 new tests in `test_alert_api_endpoints.py`, all passing. 60 total suite tests.

## What We Tried

1. Considered serverless evaluation (cron job pattern) — ruled out, adds external scheduler dependency
2. Explored rule engine libraries (Drools-style) — way too heavy, KISS won
3. Started with in-process polling loop (every 5s) — worked fine for SQLite WAL latency

## Root Cause Analysis

The complexity spike here was predictable but accepted. Anomaly detection requires stateful context (baselines, cooldowns) that doesn't fit cleanly into stateless HTTP request handling. Bundling it into the server process is the simplest path, but it means evaluation state is lost on restart. This was a deliberate YAGNI call — distributing evaluation state requires Redis/Kafka, which is premature.

The 188-line evaluator is one bad feature away from becoming a maintenance nightmare. Should have split `threshold_checker.py` out from `rolling_window_aggregator.py` from day one.

## Lessons Learned

1. **Cooldown state needs persistence**: In-memory cooldown dict survives 0 restarts. Add DB-backed cooldown table before v1.0 or silent alert storms will happen after every deploy.
2. **Missing span detection is speculative**: No formal span schema = detection based on frequency heuristics. Document this limitation explicitly in API docs before users rely on it.
3. **Evaluator is already oversized**: Split `alert_evaluator.py` into `threshold_checker.py` + `baseline_calculator.py` before next feature touches it.
4. **10s badge polling is noisy**: 60 clients * 10s polling = 6 req/s for alert count. Replace with SSE counter push or WebSocket at scale.

## Next Steps

- Add DB-backed cooldown persistence (before v1.0)
- Document `missing_span` heuristics in API reference
- Modularize `alert_evaluator.py` when next feature touches it
- Replace sidebar badge polling with SSE push event

# Multi-Tenant Auth: JWT, API Keys, Per-User Isolation

**Date**: 2026-02-28 23:15
**Severity**: Critical (Security boundary, data isolation)
**Component**: Server auth layer, storage isolation, SSE, Dashboard
**Status**: Resolved

## What Happened

Shipped full multi-tenant auth: user registration, JWT login (HS256, 24h), API key management (SHA-256 hashed, `al_` prefix), per-user data isolation across traces/alerts, SSE per-user filtering, orphan data migration to admin on startup, and dashboard auth UI. 2,559 lines added across 39 files. 27 new tests, 86 total, 86% coverage.

## The Brutal Truth

This is the feature that keeps me up at night. Auth is the one thing you cannot get wrong. We went from a completely open API with zero auth to a multi-tenant JWT+API key system in a single commit. That's either impressively fast or terrifyingly reckless depending on your perspective.

The orphan migration (`auth_seed.py` auto-assigning pre-auth data to admin on startup) is clever but also a footgun. Any user who ran AgentLens before this update now has all their traces owned by admin. If they create their own account, they get a blank slate. Their historical data is invisible to them unless they're given admin rights. No notification, no migration wizard — just silent ownership transfer. This will confuse early adopters.

The hardest part technically was SSE per-user filtering. The existing SSE implementation broadcast to all subscribers. Retrofitting user context into long-lived SSE connections meant threading JWT validation through the streaming endpoint without breaking the existing connection lifecycle. That `sse.py` rewrite (40 lines modified) was the most delicate surgery of the entire session.

## Technical Details

**New server files (commit 0434229):**
- `server/auth_models.py` (59 lines): User + APIKey SQLModel tables; password bcrypt hash; key SHA-256 hash
- `server/auth_storage.py` (113 lines): User CRUD, API key CRUD; lookup by hash; no plaintext keys stored
- `server/auth_jwt.py` (32 lines): HS256 sign/verify; 24h expiry; SECRET_KEY from env
- `server/auth_routes.py` (91 lines): POST /auth/register, /auth/login, /auth/api-keys CRUD
- `server/auth_deps.py` (59 lines): FastAPI dependencies; Bearer token + X-API-Key header; returns User
- `server/auth_seed.py` (55 lines): Admin bootstrap; orphan migration on startup

**Modified server files:**
- `server/main.py` (+62/-7 lines): Auth router mounted; startup seed; middleware order matters
- `server/storage.py` (+37 lines): `owner_id` filter on all trace queries
- `server/alert_storage.py` (+38 lines): `owner_id` filter on all alert queries
- `server/sse.py` (+40/-1 lines): Per-user subscriber registry; JWT validation on connect
- `server/requirements.txt`: Added `python-jose[cryptography]`, `bcrypt`

**New dashboard files:**
- `dashboard/src/lib/auth-context.tsx` (60 lines): React context; token storage in localStorage; auto-refresh check
- `dashboard/src/lib/auth-api-client.ts` (73 lines): login/register/api-key API calls
- `dashboard/src/lib/fetch-with-auth.ts` (27 lines): Wraps fetch; injects Bearer token; 401 → redirect to login
- `dashboard/src/pages/login-page.tsx` (89 lines): Login + register tabs
- `dashboard/src/pages/api-keys-page.tsx` (130 lines): Create/list/revoke API keys

**Test results:**
```
86 passed in 12.3s (86% coverage)
New: test_auth.py (125 lines, 15 cases) + test_tenant_isolation.py (177 lines, 12 cases)
```

**Key security decisions:**
- SECRET_KEY mandatory env var — server refuses to start without it (fail-safe)
- API keys shown once on creation, SHA-256 hashed in DB, never retrievable
- bcrypt for passwords (cost factor 12)
- No refresh tokens yet — 24h expiry, re-login required

## What We Tried

1. Considered OAuth2/OIDC first — too heavy for self-hosted single-org deployment, ruled out
2. Looked at cookie-based sessions — SSE connections don't handle cookie rotation cleanly; Bearer token cleaner
3. Evaluated storing API keys plaintext with encryption-at-rest — rejected; SHA-256 hash is irreversible and simpler

## Root Cause Analysis

The 39-file scope of this commit is alarming in retrospect. Auth retrofitted onto an existing system always touches everything — every query, every route, every test fixture needed an `owner_id` or auth header. The breadth was unavoidable, but the depth of individual files stayed manageable (none exceeded 200 lines).

The test count jump from 60 to 86 reflects how much surface area auth adds. The `conftest.py` expansion (+29 lines) was the clearest signal that the test harness architecture was stretched — fixture composition got complex when you need both `authed_client` and `admin_client` and `other_user_client` variants.

## Lessons Learned

1. **Orphan migration is a silent UX break**: Early users lose access to their historical data without warning. Add a migration notice or per-user adoption flow before public launch.
2. **SSE auth is genuinely hard**: Long-lived connections + short-lived tokens = edge cases. The current impl validates token on connect but not on reconnect. If token expires mid-stream, client gets stale events. Need 401 event type + client-side token refresh on SSE error.
3. **SECRET_KEY in env is right but undocumented**: `.env.example` doesn't exist yet. New deployers will hit a cryptic startup error. Add it.
4. **Tenant isolation tests were the highest-value tests written**: `test_tenant_isolation.py` found 2 storage-layer bugs during development that would have silently leaked data across tenants. Write isolation tests first next time, not last.
5. **39 files in one commit is too big**: Should have split auth backend (phase 1) from dashboard UI (phase 2) from test coverage (phase 3). Easier to review, easier to bisect if broken.

## Next Steps

- Add `.env.example` with `SECRET_KEY` documented
- Implement SSE `401` event + client token refresh on stream reconnect
- Add user-visible migration notice for pre-auth data assigned to admin
- Evaluate refresh token flow for sessions > 24h (enterprise use case)
- v0.5.0 tag all roadmap items complete — project enters maintenance + growth mode

## Unresolved Questions

- Should we support LDAP/SSO for enterprise self-hosted deploys?
- Refresh tokens: stateless (JWT) or stateful (DB table)? Stateless simpler but can't revoke without SECRET_KEY rotation.
- API key scopes: currently all-or-nothing. Need read-only keys for monitoring integrations?
- Multi-org support (multiple orgs per instance) vs. single-org per deploy? Current model is single-org.
